## PR Review: MushroomDAO/CometENS#5 — feat: Milestone B/C/D — NFT subdomains, plugin arch, proof mode

**Verdict: REQUEST_CHANGES** (3 new findings from adversarial PK review)

### Findings

**[PK-added] High** `contracts/src/L2RecordsV3.sol:436` — One-wallet-one-subdomain limit not enforced on-chain
- `_registerNode()` skips writing `_primaryNode[newOwner]` but still mints a new NFT
- Any active registrar or contract owner can call `registerSubnode()` multiple times for the same `newOwner`, bypassing the anti-abuse constraint
- The previous check (`primaryNode != 0`) was removed but not replaced with an enforced on-chain limit
- Fix: add `require(_primaryNode[newOwner] == 0, "already registered")` in `_registerNode()` BEFORE minting the NFT

**[PK-added] Medium** `workers/api/src/index.ts:257` — Rate limiting before auth can DoS arbitrary users
- `checkRateLimit()` runs BEFORE signature verification and keys on unauthenticated `payload.from`
- An attacker can spam requests with a victim's address and invalid signatures, causing legitimate requests to be 429-blocked
- Fix: move `checkRateLimit()` to after signature verification passes, or use IP-based rate limiting for unauthenticated requests

**[PK-added] Medium** `workers/api/src/index.ts:203` — Reverse lookup stale after transfer
- `/lookup` trusts cached `reg:<address>` label as long as `primaryNode(address)` is non-zero
- NFT transfer does not clear sender's `primaryNode`, so old owners still resolve to sold names
- Fix: either add a `_beforeTokenTransfer` hook that clears `_primaryNode[from]`, or invalidate the `reg:` KV entry on transfer
